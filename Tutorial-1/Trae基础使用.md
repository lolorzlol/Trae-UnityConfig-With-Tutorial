## 计划模式、构建模式、Solo模式

### 模式概览

```
┌─────────────────────────────────────────────────────────┐
│                    Trae 双重开发模式                      │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   IDE 模式                                    Solo 模式 │
│   ┌─────────────────┐                         ┌─────────┐│
│   │  Chat 模式      │                         │         ││
│   │  智能问答辅助    │                         │ 自主开发 ││
│   │  代码解释/修复   │                         │ 全流程  ││
│   ├─────────────────┤                         │ 代理    ││
│   │  Builder 模式   │                         │         ││
│   │  项目自动生成    │                         │         ││
│   │  需求→代码      │                         │         ││
│   └─────────────────┘                         └─────────┘│
│                                                         │
│   开发者控制流程               AI自主完成全流程           │
└─────────────────────────────────────────────────────────┘
```

### Chat 模式（智能问答）

**适用场景**：日常编码辅助、代码解释、Bug 修复

**核心能力**：
- 上下文锁定：AI 理解整个项目上下文
- 多模态交互：截图上传分析架构图、异常堆栈
- 代码生成：自然语言描述生成代码

**使用方式**：
- 快捷键：`Ctrl+U`（Windows）/ `Command+U`（macOS）
- 点击右上角聊天图标

### Builder 模式（项目构建）

**适用场景**：从零创建项目、快速原型开发

**核心能力**：
- 自然语言描述需求 → AI 自动生成完整项目结构
- 自动安装依赖、配置环境
- 对话迭代优化

**操作流程**：
```
1. 点击右侧「Builder」标签
      ↓
2. 输入需求描述（如"创建 Python 贪吃蛇游戏"）
      ↓
3. AI 生成代码文件、依赖清单、资源文件
      ↓
4. 点击「运行」自动安装依赖
      ↓
5. 对话迭代优化
```

### Solo 模式（自主开发代理）

**适用场景**：全流程自主开发、复杂项目实现

**核心特点**：
- AI 作为自主开发代理，独立完成从需求分析到部署的全流程
- 适合开发者完全放手让 AI 主导的场景

| IDE 模式 | Solo 模式 |
|----------|-----------|
| 开发者控制关键流程 | AI 自主完成全流程 |
| 实时建议和补全 | 自然语言需求 → 完整项目 |
| 适合日常开发迭代 | 适合快速原型/新项目 |

## 子代理创建

### 智能体架构

Trae 采用**多智能体协同架构**，将传统 IDE 从"工具集合"升级为"AI 团队协作平台"。

```
用户请求
    ↓
┌─────────────────────────────────────┐
│           主代理 (Main Agent)        │
│         任务分析和分配协调            │
└─────────────────────────────────────┘
    ↓
    ├── 子代理 1 (代码生成)
    ├── 子代理 2 (代码审查)
    ├── 子代理 3 (测试执行)
    └── 子代理 4 (Bug 修复)
    ↓
整合结果 → 输出给用户
```

### 子代理类型示例

| 子代理类型 | 职责 | 可用工具 |
|------------|------|----------|
| **Explore Agent** | 代码探索和结构分析 | Read, Grep（只读） |
| **Code Agent** | 代码生成和修改 | Write, Edit |
| **Test Agent** | 测试执行和验证 | Bash, Test Runner |
| **Review Agent** | 代码审查和质量检查 | Read, Grep |

### 创建自定义子代理

通过 Skills 配置创建多角色子代理：

```
.trae/
├── agents/
│   └── code-reviewer.md      ← 子代理定义
└── skills/
    ├── frontend-patterns/     ← 前端技能
    ├── backend-patterns/      ← 后端技能
    └── security-review/       ← 安全审查技能
```

**子代理定义示例**（code-reviewer.md）：
```markdown
---
name: code-reviewer
description: 代码审查专家，关注代码质量和安全性
tools:
  - Read
  - Grep
---

# 代码审查规范
- 检查代码风格一致性
- 识别潜在安全漏洞
- 建议性能优化点
```

# 一、markdown文档是AI Agent的氧气

## 1.1 为什么需要配置文件？

- AI模型没有"记忆"，每次对话都是新的开始
- 配置文件 = AI的"长期记忆"和"行为准则"
- 好的配置让AI从"聊天机器人"变成"懂你的编程伙伴"

## 1.2 配置文件的本质

| 没有配置 | 有配置 |
|---------|--------|
| 每次重新解释需求 | AI自动理解项目规范 |
| 结果不可预测 | 输出风格一致 |
| 无法复用经验 | 最佳实践自动应用 |

# 二、AGENTS.md文件介绍

## 2.1 什么是AGENTS.md？

- **统一标准**：2025年5月由AMP团队提出，旨在统一分散的Agent工具规则
- **跨平台支持**：OpenAI Codex、Anthropic Claude、Google Gemini CLI
- **项目级配置**：放在项目根目录，自动加载


## 2.3 AGENTS.md加载机制

```
项目根目录/AGENTS.md
    ↓
子目录/AGENTS.md (可覆盖)
    ↓
全局fallback (~/.claude/)
```

# 三、一个例子：让AI只回复中文

## 3.1 最简AGENTS.md示例

```markdown
# 项目规范

## 语言要求
- 所有回复必须使用中文
- 代码注释使用中文
- 变量命名可使用英文，但需在注释中说明含义

## 代码风格
- 使用4空格缩进
- 函数必须有文档字符串
```

## 3.2 放置位置

```
my-project/
├── AGENTS.md          ← 放在这里
├── src/
├── tests/
└── README.md
```

## 3.3 效果验证

- 启动AI对话，AI会自动读取并遵循规则
- 无需每次重复说明需求

# 四、Anthropic六大核心工作流规则

## 4.1 Plan Mode Orchestration（计划模式编排）

| 原则 | 说明 |
|------|------|
| 非平凡任务先规划 | 3+步骤或架构决策必须进入计划模式 |
| 出问题就停下来 | 不要硬撑，停下来重新规划 |
| 详细规格输入 | 减少歧义，使用编号步骤 |

```
任务 → 分析复杂度 → 简单任务直接执行
                  → 复杂任务 → 进入Plan Mode → 写计划 → 执行
```

## 4.2 Convert List to Subagents（子代理分工）

- **大量使用子代理**保持主上下文干净
- 一个任务一个子代理，专注执行
- 研究、探索、并行分析外包给子代理

```
主Agent (协调者)
    ├── 子Agent 1 (研究)
    ├── 子Agent 2 (代码审查)
    └── 子Agent 3 (测试)
```

## 4.3 Self-Improvement Loop（自我改进循环）

```
用户纠正
    ↓
更新 tasks/lessons.md
    ↓
写规则防止重复错误
    ↓
验证改进效果
    ↓
应用到相关项目
```

## 4.4 Verification Defaults On（验证默认开启）

- **证明能用才算完成**，不靠猜测
- 对比主分支与你的改动
- 问自己："高级工程师会认可这个吗？"
- 做断言、日志、测试套件修正

## 4.5 Elegant Elegance（优雅平衡）

| 场景 | 行动 |
|------|------|
| 非平凡改动 | 暂停问："有没有更优雅的方式？" |
| 补丁式修复 | "既然我已了解全貌，实现一个优雅的方案" |
| 简单明显修复 | 跳过，不要过度工程化 |

## 4.6 Autonomous Bug Fixing（自主修复Bug）

- 发现Bug → 直接修复，不问用户
- 指出日志、错误、失败测试，然后解决
- 用户不需要切换上下文
- 主动修复CI失败测试

## 4.7 Task Management（任务管理）

```
1. 规划优先 → 写计划到 tasks/todo.md（带checkbox）
2. 验证计划 → 开始前确认
3. 增量实施 → 完成一步打一个勾
4. 解释变更 → 每步提供高层总结
5. 记录结果 → 在 tasks/todo.md 添加review部分
6. 记录教训 → 更新 tasks/lessons.md
```

## 4.8 Core Principles（核心原则）

| 原则 | 含义 |
|------|------|
| **Simplicity first** | 每个改动尽可能简单，影响最小代码 |
| **Don't be lazy** | 找根本原因，不用创可贴，高级工程师标准 |
| **Minimal impact** | 改动只涉及必要内容 |
